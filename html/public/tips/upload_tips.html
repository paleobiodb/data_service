Tip Sheet for Batch Uploads

This page was written by John Alroy, February 2004

General comments

Paleobiology Database staff members may perform batch uploads of pre-existing data files in some cases. We prefer not to do so unless it is necessary. Our criteria for agreeing to an upload are discussed on the FAQ page. Our reasons for avoiding uploads include:

  • Corrupt data. All of our normal JavaScript and CGI script tests on data are circumvented by a batch upload. We have no easy way to confirm that data values are correct and mandatory fields are included other than eyeballing the data. Inevitably, some of the data that go in do not meet PBDB standards.
  • Redundant data. Likewise, we have no simple way to confirm that the data you are uploading do not duplicate records we already have. Normally, our scripts do check for redundant reference and collection records. In the past, we have had major problems with duplicate records being batch uploaded.
  • Uploading errors. You might have done a great job preparing your data, but we might mess up ourselves. For example, occurrence records might not be matched up to collection records correctly; data fields or values might be left out accidentally; or we might make a systematic error in translating field values.
  • Our time. With all the complications listed above, we have to be extremely careful with the upload procedure, going through multiple test runs before the final try. We have to spend valuable time confirming that data values are correct, fields in the data files directly map into PBDB and fields, and fields are identified correctly. We also often have to write a custom script to do the upload. Finally, there's a lag time involved in correspondence with the authorizer during the testing phase.
  • Your time. Because an upload can take weeks, you might get the job done faster simply by hand entering the data.
  • Lack of benefits. Statistics show that the top rate of data entry on our system is about one collection every five or ten minutes, i.e., when nothing other than keystroking is involved. However, data enterers average about one collection per hour overall. So, some 80 - 90% of data entry consists of identifying, copying, and interpreting publications, not keystroking per se. Furthermore, because you can use our preferences function to preset fields, all you are saving by doing an upload is time spent navigating web pages. As a result, the savings to you of doing an upload are minimal.
  • Expense. Uploads have to be handled by the database manager, who is paid three or four times as much as a research assistant. So, we'd rather have an assistant work for a month on a manual upload than have the DBM work for two weeks on a batch upload.

Procedure

Once you decide to ask for an upload, please follow these steps.

  • Write us a letter stating the size and scope of your data set, and asking whether and when we can do the upload.
  • Format your data files correctly (see below).
  • Hand-check each reference record and collection record to make sure we don't already have the same (or a similar) record in the Database. Remove duplicates.
  • Unless you're not uploading any new references, wait for us to go through the rest of the upload procedure (see below) just for the references. Then match the new reference numbers to each of the new collection, occurrence, etc. records. Remember, every record must match to a reference number.
  • E-mail us the data files.
  • Wait for us to write back and ask you to check over the test upload data on the backup server.
  • Write back to ask for corrections or confirm that everything is okay. Repeat these steps if necessary.
  • Once you give us the go ahead, wait for us to perform the final upload and tell you it's done.

Formatting

General rules:

  • There should be a reference number column in each table giving the official PBDB reference number for each and every record.
  • Small things count. For fields that have fixed values (e.g., pulldowns), make sure there are no extra or missing spaces, quotes, periods, etc.
  • Flat files are best. We prefer ASCII formatted, tab delimited files over Excel spread sheet files.
  • Always give both your name (the authorizer) and the name of the person (or people) who actually keystroked the data (the enterer). If there is more than one enterer, you must have an enterer name column in the data file giving the name for each record in our standard format (e.g., "A. Smith").

References:

  • The first initials and last names for the first and second authors must go in separate fields. All of the other author names must go in one other field.
  • The year field must be filled with an actual year. "pers. commun." or "unpublished" is unacceptable.
  • Be very careful that if a journal name is present in the form's pulldown menu, it is spelt exactly the same way in your data file.

Collections:

  • The reference number field is mandatory, and the best approach is for us to upload the references first so you can put the official PBDB reference numbers into your data file by hand. However, another solution is for you to assign arbitrary numbers starting with 1 to your references, and then include these numbers in a column of your collections table. So, if your collection came from the 15th reference listed in your references table, put the number "15" in the reference number field of your collection record. We'll then try to figure out how those numbers correspond to what's in the reference field.
  • Be sure to spell the country name exactly as it is given in the pulldown menu. Use "United States" instead of "USA," "U.S.A.," or "United States of America."
  • The latitude/longitude fields must be split up. The degree values (i.e., the integers before the decimal point) must be in their own fields. Minutes and seconds each go in a separate field. The part after the decimal must go in a separate field; do not start those numbers with a decimal point, because its presence is assumed. The direction (e.g., "North") also goes in a separate field.
  • Make sure that exact locality information such as township and range goes only in the "Location details" field and not in any other comment field.
  • The early/middle/late pulldown value for an interval (e.g., "Late/Upper" in "Late/Upper Cambrian") must go in its own field. Only the actual name (e.g., "Cambrian") should go in the interval name field.
  • Interval names must appear exactly as they are seen in one or more of the official time scales used by the Database.
  • Be particularly careful to include double quotes for lithology names like "shale" or "limestone" that need them.
  • It is common to include information in the taphonomy and collecting information sections on "Modes of preservation" (usually "body"), "Assemblage components" (often "macrofossils"), and "Collection types" (usually "field collection").

Occurrences:

  • The reference numbers are handled as with reference numbers in the collections table.
  • The collection number field is mandatory but we can't compute it until the collections are assigned numbers. So, give the collections sequential, arbitrary numbers and place the appropriate numbers in a collection number field.
  • Put modifiers (e.g., "cf.") and actual names (e.g., "Equus") in separate fields.
  • If you're going to use informal names, make sure to put "informal" in the modifier field.
  • Always correctly capitalize genus and species names. Never use all caps, for example.
  • Never include any spaces, quotes, parentheses, etc. in a genus/higher taxon or species name field. A name is a name, period.
  • Speaking of periods, always include the period in "indet." or "sp."
  • Make sure to include the abundance units field if you are including abundance values.
  • Correct spelling of names is absolutely crucial. Make a list of all the distinct taxon names appearing in your data, alphabetize it, and hand-check for spelling variants and errors.
